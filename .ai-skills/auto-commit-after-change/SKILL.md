# 自动提交代码规则

当完成任何代码修改后，若工作区存在未提交变更，必须自动创建一次 git 提交，以保证修改随时可追溯并避免丢失。

## 触发条件

满足以下任一条件时触发：

1. 完成功能开发 / Bug 修复 / 代码重构
2. 新增或修改测试、构建、配置、脚本
3. 用户明确表达“改完就提交”“自动提交”“每次修改都提交”

## 执行步骤

### 1. 检查是否有变更

```bash
git status --porcelain=v1 -b
```

如果没有任何文件变更，跳过后续步骤。

### 2. 暂存变更

```bash
git add -A
```

### 3. 生成提交信息并提交

提交信息要求：
- 一行摘要，使用 Conventional Commits 风格：`feat:` / `fix:` / `refactor:` / `chore:` / `docs:` / `test:`
- 摘要应描述本次修改的主要目的，避免包含任何敏感信息（token、密钥、账号、内部地址等）

```bash
git commit -m "<type>: <summary>"
```

### 4. 提交后确认状态

```bash
git status --porcelain=v1 -b
```

## 注意事项

- 不要提交包含密钥、token、证书等敏感文件；如发现疑似敏感信息，先移除并更新 .gitignore 再提交
- 如果存在明显的构建/类型错误，应先修复再提交
- 避免空提交（没有变更时不提交）

## 自动触发时机

当对话中完成了“涉及代码修改的任务”且工作区存在未提交变更时自动触发。
